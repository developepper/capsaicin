# T06: Implement a second backend adapter (OpenAI Codex CLI)

## Goal

Add a concrete BaseAdapter implementation for a second backend (OpenAI Codex CLI or similar) that follows the same RunRequest/RunResult contract, proving the adapter model works for non-Claude backends.

## Validated Local Evidence

The following has already been validated locally and should be treated as
known-good discovery for this ticket:

- `codex` binary was found on `PATH` at `/opt/homebrew/bin/codex`
- `codex --version` reported `codex-cli 0.116.0`
- `codex --help` showed top-level non-interactive subcommands including
  `exec` and `review`
- `codex exec --help` confirmed support for:
  `--json`, `--output-schema <FILE>`, `--model`, `--sandbox`,
  `--ephemeral`, and `--cd`
- `codex review --help` confirmed support for:
  `--base <BRANCH>`, `--uncommitted`, and `--commit <SHA>`
- `codex login status` reported `Logged in using ChatGPT`

Observed transport behavior from a local `codex exec --json` probe:

- stdout is JSONL event output rather than a single final JSON envelope
- observed event types included:
  `thread.started`, `turn.started`, `error`, and `item.completed`
- Codex attempted websocket transport first and then fell back to HTTPS
- in the restricted environment used for discovery, the run failed due to
  network resolution/transport errors rather than prompt execution errors

Observed successful and failure-path behavior from direct CLI evidence:

- `codex exec --json --sandbox read-only --ephemeral "Reply with exactly hi"`
  completed successfully with JSONL events:
  - `thread.started`
  - `turn.started`
  - `item.completed` with `item.type = "agent_message"` and `item.text = "hi"`
  - `turn.completed` with token usage metadata
- repeated successful probes showed stable shape for this trivial case, with
  only token counts varying between runs
- successful trivial `exec --json` runs exited with code `0`
- `codex exec --json --output-schema /tmp/codex-schema.json ...`
  failed with a structured `invalid_json_schema` error when the schema omitted
  `"additionalProperties": false`
- repeated schema probes reproduced the same `turn.failed` JSONL shape with
  `error.code = "invalid_json_schema"` and the same
  `"additionalProperties" is required to be supplied and to be false`
  validation message
- invalid schema runs exited with code `1`
- `codex exec --json --output-schema /tmp/codex-schema.json --sandbox read-only --ephemeral "Return verdict=pass"`
  succeeded once the schema included `"additionalProperties": false`
  and returned:
  - `thread.started`
  - `turn.started`
  - `item.completed` with `item.type = "agent_message"` and
    `item.text = "{\"verdict\":\"pass\"}"`
  - `turn.completed` with usage metadata
- `codex exec --json --sandbox read-only --ephemeral "Create a file named codex_probe.txt containing hello"`
  did not emit a distinct machine-readable permission-denied event; instead it
  returned normal `agent_message` items explaining that the filesystem was
  read-only, including one sample that explicitly said the write failed with
  `Operation not permitted`
- the read-only write-refusal sample still exited with code `0`, which means
  refusal/blocked-write behavior cannot be inferred from process exit code
  alone
- `codex review --base master "Review the current branch for bugs and missing tests"`
  failed locally with a CLI argument error stating that `--base <BRANCH>`
  cannot be used with `[PROMPT]`, even though the usage text still renders as
  `codex review --base <BRANCH> [PROMPT]`
- `codex review --base master` succeeded without a custom prompt, launched a
  repository review workflow, printed human-readable transcript output rather
  than JSONL, and exited with code `0`

Practical implication:

- the Codex adapter should be designed around `codex exec` as the general
  non-interactive invocation surface
- the adapter must not assume Claude-style single-envelope stdout
- event-stream parsing is likely required for `--json` mode

## Remaining Evidence To Capture

The following evidence is still useful before implementation details should be
considered fully locked down:

1. A transport/auth/runtime failure sample with confirmed exit code in a normal
   network-enabled environment
2. A stronger sandbox/permission-denial sample only if we still hope to map
   Codex behavior to `permission_denied`; current evidence suggests read-only
   failures may appear as ordinary assistant text instead of a distinct machine
   signal while still exiting `0`
3. A decision on whether `codex review` should be ignored entirely for adapter
   parity or supported as a separate specialized review mode, since its output
   shape is transcript-style and not aligned with the shared structured
   `RunResult` contract

## Scope

- New src/capsaicin/adapters/codex.py implementing BaseAdapter.execute()
- Subprocess invocation of the codex CLI with appropriate flags for prompt, mode, output format
- Map Codex CLI exit codes and output format to RunResult fields (exit_status, structured_result, raw_stdout, raw_stderr)
- Handle permission-denial detection analogous to claude_code.py
- Register 'codex' in the adapter registry from ticket 2
- Add adapter_metadata extraction (session info, token usage if available)

## Non-goals

- Supporting every Codex feature or flag
- Structured output validation for Codex (it may not support --json-schema; adapter should handle gracefully)
- Making Codex the default for any role
- Automatic capability detection — the operator supplies evidence about what works

## Acceptance Criteria

- CodexAdapter class exists and implements BaseAdapter.execute()
- A config with backend = 'codex' and appropriate command resolves to CodexAdapter via the registry
- RunResult fields are correctly populated from Codex CLI output including exit_status mapping
- Permission denial is detected and surfaced as exit_status = 'permission_denied'
- The adapter handles the case where structured output is not available (falls back to result_text)
- Existing Claude adapter and workflows are unaffected
- The adapter implementation is explicitly grounded in the validated Codex CLI
  invocation and output evidence captured above, rather than guessed flags or
  output assumptions

## Implementation Notes

- Study claude_code.py closely — it is the reference implementation for subprocess invocation, output parsing, metadata extraction, and error handling
- Prefer `codex exec` as the primary adapter entrypoint for implementer,
  planner, and reviewer roles unless later evidence proves `codex review`
  is materially better for implementation review and still fits the shared
  `RunRequest` / `RunResult` contract
- Codex CLI uses different flags from Claude and emits JSONL event streams in
  `--json` mode rather than a single final JSON envelope; design the adapter
  around event parsing rather than envelope parsing
- For simple successful `exec --json` runs, the final assistant text has been
  observed in an `item.completed` event where `item.type = "agent_message"` and
  the turn ends with a separate `turn.completed` usage event
- `exec --json` exit codes observed so far:
  - simple success: `0`
  - schema-validation failure: `1`
  - read-only write refusal expressed as assistant text: `0`
- The adapter should accept `model` from `adapter_config` and pass it through
  using `--model`
- If Codex supports `--output-schema` reliably, use it for reviewer/planner
  structured output; otherwise parse the final assistant text or final JSONL
  event payload and fall back to storing text in `result_text`
- Codex schema mode is stricter than the naive probe suggested: schema objects
  must include `"additionalProperties": false` at the top level or the CLI
  returns a structured `invalid_json_schema` failure
- Do not assume read-only sandbox failures map cleanly to a dedicated
  `permission_denied` machine signal; current evidence shows at least one case
  where Codex returned ordinary assistant text describing the restriction and
  still exited `0`
- Do not commit to `codex review` as the implementation-review backend until
  you explicitly choose to support its transcript-style output separately;
  current evidence shows:
  - `codex review --base master` works without a prompt and exits `0`
  - `codex review --base master "..."` fails with a CLI argument error despite
    help text implying that prompt usage is legal
  - output shape is human-readable transcript text, not JSONL
- Treat websocket/HTTPS transport failures, auth failures, and CLI runtime
  panics as distinct failure evidence in `raw_stderr` / `raw_stdout` even if
  they collapse to `exit_status = 'failure'`
- Consider making the command and common flags configurable through AdapterConfig fields so the operator can tune invocation without code changes
- If the actual Codex CLI is not available for testing, implement the adapter structure with clear TODOs for output format mapping and write a mock/stub test
