# T09: Evidence audit trail and inspectability

## Goal

Make validation evidence inspectable after the fact so a human can trace why a backend integration was implemented a certain way, linking evidence to the planning decisions and implementation runs that consumed it.

## Scope

- Record which evidence records were included in each agent run's prompt — add a join table agent_run_evidence(run_id, evidence_id) via migration
- On the agent run detail view (within ticket detail and epic detail), show which evidence was provided to that run
- On the evidence detail view (within epic detail), show which agent runs consumed that evidence record
- Add an 'Evidence Timeline' view on the epic detail page showing when evidence was added, when it was consumed by runs, and when requirements were satisfied or waived
- Include evidence references in materialized ticket docs so the downstream implementer can see what evidence informed the plan

## Non-goals

- Version history of individual evidence records (immutable create-or-delete in first pass)
- Exporting evidence to external systems
- Diffing evidence between planning cycles

## Acceptance Criteria

- Each agent run record tracks which evidence IDs were included in its prompt context
- The run detail section on ticket and epic detail pages lists the evidence records that were provided
- The evidence panel on the epic detail page shows which runs consumed each evidence record
- Materialized ticket docs include a section listing evidence references from the parent epic
- An operator can answer 'what evidence informed this implementation decision' by inspecting the UI

## Implementation Notes

- A join table agent_run_evidence(run_id, evidence_id) is cleaner than embedding IDs in a JSON column since it supports queries in both directions
- When building prompts (ticket 7 handlers), after loading evidence and before the adapter call, insert the join records
- The evidence timeline can be assembled from backend_evidence.created_at + agent_run_evidence joins + evidence_requirements status changes — render as a simple chronological list
- For materialization, extend the ticket markdown template in materialize.py to include an '## Evidence References' section listing evidence titles and types from the parent epic
- Keep the audit data lightweight — just link IDs and timestamps, not duplicated content
