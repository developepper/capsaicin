# T01: Backend validation evidence data model and migration

## Goal

Create the schema and domain types for storing structured backend validation evidence (example commands, stdout/stderr envelopes, structured-result samples, permission-denial observations, and behavioral notes) so that evidence can be attached to epics and planned tickets.

## Scope

- New migration adding backend_evidence table with columns: id, epic_id (nullable FK), planned_ticket_id (nullable FK), evidence_type enum (command, output_envelope, structured_result_sample, permission_denial, behavioral_note), title, content (JSON text), created_at, updated_at
- New migration adding evidence_requirements table: id, epic_id, planned_ticket_id (nullable), description, status enum (pending, satisfied, waived), evidence_id (nullable FK back to backend_evidence), suggested_command (nullable TEXT for exact CLI command the operator should run), created_at
- Python dataclasses in a new evidence.py module for BackendEvidence and EvidenceRequirement
- Basic CRUD query helpers: insert_evidence, list_evidence_for_epic, list_evidence_for_ticket, insert_requirement, update_requirement_status, list_requirements_for_epic

## Non-goals

- UI for capturing evidence (ticket 4)
- Using evidence in prompts (ticket 7)
- Gating logic based on missing evidence (ticket 8)

## Acceptance Criteria

- A new SQL migration creates backend_evidence and evidence_requirements tables with proper FK constraints and indexes
- BackendEvidence and EvidenceRequirement dataclasses exist with to_dict/from_dict round-trip support
- EvidenceRequirement includes a suggested_command field for storing exact CLI commands the operator should run to produce the evidence
- CRUD helpers can insert and query evidence and requirements scoped to an epic or planned ticket
- Existing tests and migrations continue to pass without regressions

## Implementation Notes

- Follow the migration numbering convention — next available is likely 0004 or 0005; check the migrations directory
- evidence_type should be a CHECK constraint enum like other status columns in the schema
- content column stores JSON so different evidence types can have different shapes (e.g., command has 'command', 'working_directory'; output_envelope has 'stdout', 'stderr', 'exit_code')
- The suggested_command field on evidence_requirements stores a ready-to-copy CLI command (e.g., 'codex --json-schema ... | head -100') that the operator can paste into a terminal to produce the needed evidence
- Use the XOR pattern from 0003 if evidence can attach to either epic or planned_ticket but not both, or allow both-nullable with a CHECK that at least one is set
- Keep the Python types in a new src/capsaicin/evidence.py module to avoid bloating adapters/types.py
