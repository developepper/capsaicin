# T07: Inject validation evidence into planning, implementation, and review prompts with suggested requirements

## Goal

Make operator-supplied backend validation evidence available in the prompts sent to all four roles — planner, planning reviewer, implementer, and implementation reviewer — so that adapter work is driven by observed evidence rather than assumptions. Also enable the planner to suggest evidence requirements with exact CLI commands the operator should run.

## Scope

- Extend build_planner_draft_prompt and build_planner_revise_prompt to accept and format evidence records into a '# Backend Validation Evidence' section
- Extend build_implementer_prompt to include relevant evidence when the ticket is part of an adapter-focused epic
- Extend build_reviewer_prompt to include relevant evidence when the ticket is part of an adapter-focused epic, so the implementation reviewer can verify that the implementation aligns with observed backend behavior
- Extend build_planning_reviewer_prompt to include evidence so the planning reviewer can verify the plan references it
- Load evidence from DB in planning_run.py (_draft_invoke_once), planning_review.py (_planning_review_invoke_once), ticket_run.py (_impl_invoke_once), and ticket_review.py (_review_invoke_once) and pass it through to prompt builders
- Format different evidence types appropriately in prompts (commands as code blocks, outputs as pre-formatted, notes as prose)
- Add planner instructions to emit suggested_evidence_requirements in PlannerResult: a list of {description, suggested_command} objects describing what CLI commands the operator should run and paste back to validate backend behavior
- In planning_run.py's _handle_draft_result, after parsing PlannerResult, persist any suggested_evidence_requirements as EvidenceRequirement rows with status=pending and suggested_command populated

## Non-goals

- Automatic evidence summarization or compression
- Filtering which evidence is relevant per ticket — include all epic-level evidence plus ticket-specific evidence

## Acceptance Criteria

- Planner draft prompts include a 'Backend Validation Evidence' section when evidence exists for the epic
- Planner revision prompts include evidence alongside the prior plan and findings
- Planning reviewer prompts include evidence so the reviewer can check plan-evidence alignment
- Implementer prompts include evidence from the parent epic when the ticket has lineage to a planned ticket with evidence
- Implementation reviewer prompts (build_reviewer_prompt) include evidence from the parent epic so the reviewer can verify the implementation matches observed backend behavior
- Each evidence type is formatted readably (commands in code blocks, outputs with stdout/stderr labeled, etc.)
- The planner can suggest evidence requirements with exact CLI commands (e.g., 'codex --help', 'codex -q "print hello" --json 2>&1') that get persisted as pending EvidenceRequirement rows with suggested_command populated
- Suggested requirements appear in the UI (ticket 4 already displays them) so the operator knows exactly what to run

## Implementation Notes

- Add a _format_evidence(evidence_list) helper in prompts.py that renders each evidence record by type
- For commands: render as shell code block with working directory noted
- For output envelopes: render stdout and stderr in labeled pre blocks with exit code
- For structured results: render as JSON code block
- For permission denials: render the denied tool/action and observed behavior
- For behavioral notes: render as quoted prose
- The evidence section should appear after the problem statement but before decomposition constraints in planner prompts
- In implementer prompts, evidence should appear after the ticket description as a 'Backend Context' section
- In reviewer prompts (build_reviewer_prompt), evidence should appear as a 'Backend Context' section so the reviewer can check that the diff's adapter behavior matches the operator-supplied evidence
- Add a new optional field suggested_evidence_requirements to PlannerResult (list of dicts with 'description' and 'suggested_command' keys). The suggested_command should be an exact, copy-pasteable CLI invocation like 'codex -q "what is 2+2" --json 2>&1 | head -50'
- In the planner system prompt, instruct: 'When the epic involves a backend you have not seen evidence for, suggest evidence requirements with exact CLI commands the operator should run.'
- Persistence of suggested_evidence_requirements happens in planning_run.py's _handle_draft_result — this is the planner runtime result-processing layer, NOT the draft_epic CLI command handler.
