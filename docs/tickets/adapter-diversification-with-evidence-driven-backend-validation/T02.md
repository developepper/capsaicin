# T02: Four-role config schema and adapter registry

## Goal

Extend config.toml to support independent backend selection for all four roles (planner, planning_reviewer, implementer, reviewer) and create an adapter registry that resolves backend name strings to adapter classes, so each loop's roles can use different backends.

## Scope

- Add [adapters.planner] and [adapters.planning_reviewer] sections to config.toml schema
- Add planner and planning_reviewer fields to the Config dataclass as AdapterConfig instances
- Backward-compatible fallback: planner defaults to implementer config, planning_reviewer defaults to reviewer config, when their sections are absent
- Create an adapter registry (dict mapping backend name string to BaseAdapter subclass) in a new src/capsaicin/adapters/registry.py
- Add a resolve_adapter(backend_name) function that returns the correct BaseAdapter subclass and raises a clear error for unknown backends
- Refactor planning_run.py to construct its adapter via registry lookup using config.planner instead of hardcoding ClaudeCodeAdapter with config.implementer
- Refactor planning_review.py to construct its adapter via registry lookup using config.planning_reviewer instead of config.reviewer
- Refactor plan_loop.py to construct planner adapter from config.planner and planning reviewer adapter from config.planning_reviewer
- Update write_default_config to include commented-out [adapters.planner] and [adapters.planning_reviewer] sections
- Update config_to_snapshot and load_config to handle the two new role slots

## Non-goals

- Implementing a second adapter (ticket 6)
- Epic-level or ticket-level overrides (ticket 3)
- Changing the implementation or review loop wiring beyond using the registry for the correct role config

## Acceptance Criteria

- Config dataclass has four AdapterConfig fields: implementer, reviewer, planner, planning_reviewer
- Config.planner defaults to implementer config and Config.planning_reviewer defaults to reviewer config when their config.toml sections are absent
- An adapter registry function resolve_adapter(backend_name) returns the correct BaseAdapter subclass
- planning_run.py uses config.planner (not config.implementer) to select and construct its adapter
- planning_review.py uses config.planning_reviewer (not config.reviewer) to select and construct its adapter
- plan_loop.py constructs its draft adapter from config.planner and its review adapter from config.planning_reviewer
- Existing configs without [adapters.planner] or [adapters.planning_reviewer] continue to work unchanged via fallback
- config_to_snapshot round-trips all four role configs correctly

## Implementation Notes

- The registry can be a simple dict in src/capsaicin/adapters/registry.py: ADAPTER_REGISTRY = {'claude-code': ClaudeCodeAdapter}
- resolve_adapter should accept the backend string and return the class (not an instance), so callers can pass command= at construction time
- planning_run.py currently does ClaudeCodeAdapter(command=config.implementer.command) — change to: AdapterCls = resolve_adapter(config.planner.backend); adapter = AdapterCls(command=config.planner.command)
- planning_review.py should use config.planning_reviewer similarly
- plan_loop.py currently creates both adapters and passes them to run_planning_loop — update to use config.planner for draft_adapter and config.planning_reviewer for review_adapter
- The fallback logic in load_config: if 'planner' not in adapters_section, copy implementer config; if 'planning_reviewer' not in adapters_section, copy reviewer config
- Keep the four roles as a constant tuple ROLES = ('implementer', 'reviewer', 'planner', 'planning_reviewer') for iteration in config logic
