# T04: UI affordances for capturing backend validation evidence with clarification workflow

## Goal

Add web UI forms and display panels on the epic detail page that let an operator create, view, and manage backend validation evidence records and evidence requirements, including a clarification workflow where the UI presents exact CLI commands to run and accepts pasted output as evidence.

## Scope

- Add an 'Evidence' panel/section to the epic detail template showing all attached evidence grouped by type
- Add a form for creating new evidence: type selector, title, and a dynamic content area (command field, stdout/stderr textareas, JSON editor for structured results, freeform text for behavioral notes)
- Add an 'Evidence Requirements' panel showing what evidence is still needed, with status badges (pending/satisfied/waived) and the suggested CLI command to run displayed in a copyable code block
- Add a 'Paste Output' shortcut flow on each pending requirement: when a suggested_command exists, show the command in a copyable code block and provide a form where the operator can paste stdout, stderr, and exit code directly — on submit, this creates an output_envelope evidence record and auto-satisfies the requirement
- Add POST routes: /epics/{epic_id}/evidence (create), /epics/{epic_id}/evidence/{evidence_id}/delete (remove)
- Add POST routes: /epics/{epic_id}/requirements (create requirement), /epics/{epic_id}/requirements/{req_id}/satisfy (link to evidence), /epics/{epic_id}/requirements/{req_id}/waive, /epics/{epic_id}/requirements/{req_id}/paste-output (create evidence from pasted CLI output and auto-satisfy)
- Extend PlanningDetailData query to include evidence and requirements lists

## Non-goals

- Inline evidence editing (create-only in first pass, delete to redo)
- Evidence on individual implementation tickets (only on epics and planned tickets)
- File upload for evidence (text input only)

## Acceptance Criteria

- The epic detail page displays all evidence records grouped by type with their content rendered readably
- An operator can create a new evidence record of each type (command, output_envelope, structured_result_sample, permission_denial, behavioral_note) via the UI form
- Evidence requirements are visible with pending/satisfied/waived status badges and, when a suggested_command exists, the exact CLI command is displayed in a copyable code block
- An operator can use the paste-output flow on a pending requirement: see the suggested command, paste stdout/stderr/exit_code, and have an evidence record created and the requirement auto-satisfied in one action
- An operator can create a requirement, satisfy it by linking evidence, or waive it
- Evidence and requirements persist across page reloads (stored in DB)

## Implementation Notes

- Follow the existing HTMX pattern — form submissions POST and redirect back to epic detail
- The paste-output route (/epics/{epic_id}/requirements/{req_id}/paste-output) should: (1) create a backend_evidence record of type output_envelope with the pasted stdout, stderr, exit_code as JSON content, (2) update the requirement to status=satisfied linking the new evidence_id, (3) redirect back to epic detail
- The suggested_command display should use a <pre><code> block with a copy-to-clipboard button or at minimum be easily selectable for copy-paste
- Group evidence display: Commands section, Output Envelopes section, etc. — each with expandable/collapsible content
- Evidence requirements can be auto-created by the planner (ticket 7) or manually by the operator here
- Keep the template additions in a new partial (e.g., partials/evidence_panel.html) included from epic_content.html
