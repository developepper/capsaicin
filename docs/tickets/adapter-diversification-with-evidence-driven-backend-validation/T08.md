# T08: Missing-evidence gating and blocker states with clarification prompts

## Goal

Surface missing or insufficient backend validation evidence as explicit blocker/human-gate states in the planning workflow, and as reviewer findings in the implementation workflow, preventing the system from planning adapter work based on assumptions while allowing implementation to proceed with appropriate warnings.

## Scope

- Add evidence completeness check: before a planning draft or planning review run, verify that all evidence_requirements with status='pending' are flagged
- When pending requirements exist at planning time, transition the epic to human-gate with gate_reason='missing_evidence' instead of proceeding with the planner or planning reviewer agent run
- Add gate_reason='missing_evidence' to the planned_epics CHECK constraint via a new migration
- Add a GateDisplayInfo entry for missing_evidence in gate_display.py with display_text that tells the operator to supply the required evidence and is_warning=True
- Display a prominent missing-evidence banner on the epic detail page listing each unsatisfied requirement with its description and, when available, the exact suggested CLI command to run in a copyable code block
- Allow the operator to waive individual requirements to unblock if they choose
- Add a 'Missing Evidence' finding category that the planning reviewer can emit when the plan references behaviors not backed by evidence
- For the implementation loop: when a ticket's parent epic has pending evidence requirements, inject a warning into the implementer prompt noting unresolved evidence, and instruct the implementation reviewer to emit a 'missing_evidence' finding if the implementation relies on unverified backend behavior

## Non-goals

- Hard-blocking the implementation loop on missing evidence — implementation surfaces it as findings/warnings instead
- Blocking the entire system — only adapter-focused epics with evidence requirements are gated at planning time
- Automatically determining what evidence is needed — requirements are operator-created or planner-suggested
- Hard-blocking waived requirements — waiving is an explicit operator override

## Acceptance Criteria

- An epic with pending (unsatisfied, un-waived) evidence requirements cannot proceed to planning draft or planning review — it transitions to human-gate with gate_reason='missing_evidence'
- The epic detail page shows a prominent banner listing missing evidence requirements, each with its description and suggested CLI command (if present) in a copyable code block
- The gate display text for missing_evidence clearly tells the operator: 'Backend validation evidence is required before planning can proceed. Run the suggested commands below, then paste results into the evidence panel.'
- Waiving a requirement removes it as a blocker and allows the planning workflow to proceed
- The planning reviewer can emit findings with category='missing_evidence' when plan content is not backed by supplied evidence
- The gate_reason='missing_evidence' is added to the planned_epics CHECK constraint and handled by the state machine
- When an implementation ticket's parent epic has pending evidence requirements, the implementer prompt includes a warning about unresolved evidence
- The implementation reviewer is instructed to emit 'missing_evidence' findings when the implementation relies on backend behavior that is not backed by evidence

- The gate_reason CHECK constraint on planned_epics needs to be extended — this requires a migration that recreates the table or uses ALTER TABLE to drop and re-add the constraint (SQLite limitation)
- The evidence check should happen in planning_run.py (before invoking the planner adapter) and planning_review.py (before invoking the planning reviewer adapter), NOT in the draft_epic CLI command handler
- Add a helper like check_evidence_completeness(conn, epic_id) -> list[EvidenceRequirement] that returns unsatisfied requirements
- The missing-evidence banner should render each pending requirement as: description text, then if suggested_command is present a <pre><code> block with the command and a note like 'Run this command and paste the output below'
- The banner should link to or be adjacent to the paste-output form from ticket 4 so the operator flow is: see missing requirement → copy command → run in terminal → paste output → requirement auto-satisfied → resume
- In gate_display.py, add: GateDisplayInfo('Backend validation evidence is required before planning can proceed. Run the suggested commands below and paste the results into the evidence panel.', is_warning=True)
- The planning reviewer prompt (from ticket 7) already includes evidence — add an instruction telling the reviewer to flag plan elements that reference unverified backend behavior with category='missing_evidence'
- For the implementation path: in ticket_run.py's _impl_invoke_once, check for pending requirements on the parent epic and append a warning section to the implementer prompt. In ticket_review.py, add reviewer instructions to flag missing_evidence. This is a softer gate than planning — it surfaces the issue without hard-blocking.
