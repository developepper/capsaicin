# T03: Role override storage and adapter resolution service

## Goal

Add epic-level and ticket-level backend override storage and a deterministic adapter resolution service that selects the backend for a given role by walking a clear precedence chain: ticket override → epic override → project config.toml → built-in fallback. Epic-level overrides are scoped to planner and planning_reviewer roles; ticket-level overrides are scoped to implementer and reviewer roles.

## Scope

- New migration adding role_overrides table: id, epic_id (nullable FK), ticket_id (nullable FK), role TEXT (one of implementer, reviewer, planner, planning_reviewer), backend TEXT, command TEXT, model TEXT (nullable), allowed_tools TEXT (nullable JSON array), created_at — with a UNIQUE constraint on (epic_id, ticket_id, role) and a CHECK ensuring exactly one of epic_id/ticket_id is set
- Add a CHECK constraint enforcing role scoping: epic_id overrides must have role IN ('planner', 'planning_reviewer') and ticket_id overrides must have role IN ('implementer', 'reviewer')
- Python dataclass RoleOverride in a new src/capsaicin/role_resolution.py with to_dict/from_dict
- CRUD helpers: set_override, get_overrides_for_epic, get_overrides_for_ticket, delete_override
- A resolve_adapter_config(role, config, epic_id=None, ticket_id=None, conn=None) function that implements the precedence chain: (1) for implementer/reviewer roles, check ticket_id override if provided, (2) for planner/planning_reviewer roles, check epic_id override if provided, (3) return getattr(config, role) from project config, (4) built-in fallback to implementer config for planner and reviewer config for planning_reviewer
- A resolve_all_roles(config, epic_id=None, ticket_id=None, conn=None) convenience function that returns a dict of role → (AdapterConfig, source_label) for display in the UI
- Refactor planning_run.py and plan_loop.py to use resolve_adapter_config(role='planner', epic_id=epic_id) to select the planner adapter
- Refactor planning_review.py and plan_loop.py to use resolve_adapter_config(role='planning_reviewer', epic_id=epic_id) to select the planning reviewer adapter
- Refactor ticket_run.py and run_ticket.py CLI command to use resolve_adapter_config(role='implementer', ticket_id=ticket_id, epic_id=epic_id) to select the implementer adapter
- Refactor ticket_review.py and review_ticket.py CLI command to use resolve_adapter_config(role='reviewer', ticket_id=ticket_id, epic_id=epic_id) to select the reviewer adapter
- Refactor loop.py to use resolve_adapter_config for both implementer and reviewer roles when constructing adapters
- Unit tests for the resolution precedence chain covering: override at each level, fallback when no override exists, role-scope enforcement rejecting invalid combinations

## Non-goals

- UI for editing overrides (ticket 5)
- Automatic override suggestion — overrides are manually set by the operator
- Planned-ticket-level overrides — only epic and implementation ticket levels in this pass
- Allowing epic-level overrides for implementer/reviewer or ticket-level overrides for planner/planning_reviewer

## Acceptance Criteria

- A role_overrides table exists with proper constraints ensuring exactly one scope (epic or ticket) is set per row
- A CHECK constraint enforces that epic_id overrides can only set planner or planning_reviewer roles, and ticket_id overrides can only set implementer or reviewer roles
- resolve_adapter_config returns the correct AdapterConfig by walking the precedence chain appropriate to the role: ticket override for implementer/reviewer, epic override for planner/planning_reviewer, then project config, then fallback
- When a ticket has an override for 'implementer' but not 'reviewer', the implementer resolves to the override and the reviewer resolves to the project config level
- planning_run.py uses resolve_adapter_config(role='planner', epic_id=epic_id) to select its adapter
- planning_review.py uses resolve_adapter_config(role='planning_reviewer', epic_id=epic_id) to select its adapter
- ticket_run.py and run_ticket.py use resolve_adapter_config(role='implementer', ticket_id=ticket_id, epic_id=epic_id) to select the implementer adapter
- ticket_review.py and review_ticket.py use resolve_adapter_config(role='reviewer', ticket_id=ticket_id, epic_id=epic_id) to select the reviewer adapter
- loop.py uses resolve_adapter_config for both implementer and reviewer roles
- resolve_all_roles returns both AdapterConfig and source label (e.g., 'epic override', 'project config', 'fallback') for each role
- Unit tests verify precedence and role-scope enforcement: epic override only accepts planner/planning_reviewer, ticket override only accepts implementer/reviewer, correct fallback behavior

## Implementation Notes

- The precedence chain is role-aware: for planner/planning_reviewer, check epic_id override then config; for implementer/reviewer, check ticket_id override then config. There is no cross-level lookup (no epic-level implementer overrides, no ticket-level planner overrides).
- The role-scope CHECK constraint in SQL: CHECK((epic_id IS NOT NULL AND role IN ('planner', 'planning_reviewer')) OR (ticket_id IS NOT NULL AND role IN ('implementer', 'reviewer')))
- resolve_adapter_config should return an AdapterConfig instance so callers can use it uniformly with resolve_adapter(result.backend) from the registry
- The conn parameter should be optional — when None, skip DB lookups and just use config (useful in tests and CLI-only mode)
- Store allowed_tools as a JSON array string in the override table, parse on read
- The reviewer adapter for implementation tickets is selected in ticket_review.py and review_ticket.py — NOT in ticket_run.py. ticket_run.py only handles the implementer role.
- loop.py constructs both impl_adapter and review_adapter separately — update both construction sites
- resolve_all_roles should return a dict like {'implementer': (AdapterConfig, 'epic override'), 'reviewer': (AdapterConfig, 'project config'), ...} for display in ticket 5's UI
