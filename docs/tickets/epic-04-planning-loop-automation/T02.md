# T02: Planner And Planning-Reviewer Contracts

## Goal

Define the run envelopes, prompt builders, schemas, and validation rules for
planner and planning-reviewer passes so planning can be executed with the same
contract discipline as the implementation loop.

## Scope

- define planner request and result structures
- define planning-reviewer structured result schema and semantic validation
- add prompt builders for planning draft, planning revise, and planning review
- decide how planning artifacts are represented inside adapter requests
- document planner/reviewer role instructions and anti-bias constraints

## Non-goals

- DB schema creation beyond what is needed to persist the new contracts
- CLI command execution flow
- loop orchestration
- approval and ticket materialization

## Acceptance Criteria

- planner prompt assembly captures the problem statement, current plan draft,
  prior planning findings, and explicit decomposition constraints
- the planner result has a concrete structured schema, sufficient to persist an
  epic draft and planned tickets without scraping prose
- planning-reviewer output is schema-constrained and semantically validated in
  the same spirit as code review results
- the contracts support reviewer findings against both epic-level and
  ticket-level planning artifacts
- the contracts make it possible for a planner revision pass to receive prior
  findings without hidden chat history
- docs and code clearly separate planning review from implementation review
  even when they share adapter plumbing
- tests cover semantic validation failures, not just nominal schema shape

## Implementation Notes

- Reuse the existing run-envelope approach instead of inventing a separate
  transport model for planning.
- Planning review findings should be rich enough to drive revision and later
  ticket materialization. At minimum, findings need scope targeting, severity,
  description, and stable identity/fingerprinting semantics.
- The planner contract should explicitly require ticket artifacts to carry
  references, dependencies, and implementation notes; do not leave those as
  optional prose if the generated implementation tickets depend on them later.
- The expected planner result should be a structured plan artifact along these
  lines:
  - plan title and problem statement summary
  - epic summary and success outcome
  - planned tickets with title, goal, scope, non-goals, acceptance criteria,
    dependencies, references, and implementation notes
  - optional sequencing notes and open questions for human review
- The planning-reviewer result should mirror implementation review shape:
  `verdict`, `confidence`, `findings`, and `scope_reviewed`, but findings must
  be able to target either the overall plan or a specific planned ticket.
