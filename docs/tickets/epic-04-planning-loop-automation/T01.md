# T01: Planning Data Model And State Machine Foundation

## Goal

Introduce the canonical planning-domain schema, state-machine rules, and query
primitives needed to support a local planning loop alongside the existing
implementation loop.

## Scope

- define planning entities and relationships in code and docs
- add migrations for planning records, planning findings, planning decisions,
  and planning transitions
- decide and implement the canonical orchestrator persistence model for
  planning runs
- define lineage storage between planning records and materialized
  implementation tickets
- add planning status enums and transition validation
- add low-level query helpers and shared IDs needed by later planning-loop
  services
- keep manual implementation-ticket creation working unchanged

## Non-goals

- planner prompt assembly
- planner/reviewer subprocess invocation
- CLI command wiring
- ticket materialization rendering
- UI pages

## Acceptance Criteria

- the schema introduces first-class planning records for at least:
  - a problem statement or planning brief
  - a planned epic
  - planned implementation tickets
  - planning review findings
  - planning decisions and transition history
- planning statuses and transition rules are defined and documented in
  [docs/state-machine.md](../../state-machine.md), including:
  `planning/new -> planning/drafting -> planning/in-review -> planning/revise -> planning/human-gate -> planning/approved` plus `planning/blocked`
- the migration resolves orchestrator persistence for planning explicitly:
  either generalize `orchestrator_state` safely for both loops or add a sibling
  planning-specific orchestrator table
- planning findings are stored in planning-specific tables rather than
  overloading the existing implementation `findings` table
- lineage required for later materialization is persisted, either via planning
  IDs on materialized implementation tickets or via an explicit join table
- migrations are additive and preserve existing implementation-loop projects
- foreign keys, indexes, and IDs are defined with the same rigor as the
  existing implementation-loop schema
- shared query helpers exist so later tickets do not need to hand-roll raw SQL
  for basic planning record loading
- docs are updated where the planning model is now concrete rather than merely
  aspirational, including the planning note in
  [docs/data-model.md](../../data-model.md)

## Design Decisions

### Planning Tables

Introduce 6 new tables. The problem statement is a column on `planned_epics`
rather than a separate entity — one problem statement produces one planning
attempt, and a separate table adds a join without adding value.

```sql
CREATE TABLE planned_epics (
    id                TEXT PRIMARY KEY,
    project_id        TEXT NOT NULL REFERENCES projects(id),
    problem_statement TEXT NOT NULL,
    title             TEXT,                -- null until first draft
    summary           TEXT,                -- null until first draft
    success_outcome   TEXT,                -- null until first draft
    sequencing_notes  TEXT,
    current_cycle     INTEGER NOT NULL DEFAULT 0,
    current_draft_attempt  INTEGER NOT NULL DEFAULT 1,
    current_review_attempt INTEGER NOT NULL DEFAULT 1,
    blocked_reason    TEXT,
    gate_reason       TEXT CHECK (gate_reason IN (
                          'review_passed','reviewer_escalated','cycle_limit',
                          'draft_failure','human_requested','low_confidence_pass'
                      )),
    status            TEXT NOT NULL DEFAULT 'new'
                      CHECK (status IN (
                          'new','drafting','in-review',
                          'revise','human-gate','approved','blocked'
                      )),
    materialized_path TEXT,
    status_changed_at TEXT NOT NULL DEFAULT (datetime('now')),
    created_at        TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at        TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE planned_tickets (
    id                TEXT PRIMARY KEY,
    epic_id           TEXT NOT NULL REFERENCES planned_epics(id),
    sequence          INTEGER NOT NULL,
    title             TEXT NOT NULL,
    goal              TEXT NOT NULL,
    scope             TEXT NOT NULL,       -- JSON array of strings
    non_goals         TEXT NOT NULL,       -- JSON array of strings
    references_       TEXT NOT NULL,       -- JSON array of strings
    implementation_notes TEXT NOT NULL,    -- JSON array of strings
    created_at        TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at        TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE (epic_id, sequence)
);

CREATE TABLE planned_ticket_criteria (
    id                TEXT PRIMARY KEY,
    planned_ticket_id TEXT NOT NULL REFERENCES planned_tickets(id),
    description       TEXT NOT NULL
);

CREATE TABLE planned_ticket_dependencies (
    planned_ticket_id TEXT NOT NULL REFERENCES planned_tickets(id),
    depends_on_id     TEXT NOT NULL REFERENCES planned_tickets(id),
    PRIMARY KEY (planned_ticket_id, depends_on_id),
    CHECK (planned_ticket_id != depends_on_id)
);

CREATE TABLE planning_findings (
    id              TEXT PRIMARY KEY,
    run_id          TEXT NOT NULL REFERENCES agent_runs(id),
    epic_id         TEXT NOT NULL REFERENCES planned_epics(id),
    planned_ticket_id TEXT REFERENCES planned_tickets(id),  -- null = epic-level
    severity        TEXT NOT NULL CHECK (severity IN ('blocking','warning','info')),
    category        TEXT NOT NULL,
    description     TEXT NOT NULL,
    fingerprint     TEXT NOT NULL,
    disposition     TEXT NOT NULL DEFAULT 'open'
                    CHECK (disposition IN ('open','fixed','wont_fix','disputed')),
    resolved_in_run TEXT REFERENCES agent_runs(id),
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE materialization_hashes (
    epic_id           TEXT NOT NULL REFERENCES planned_epics(id),
    planned_ticket_id TEXT REFERENCES planned_tickets(id),  -- null = epic-level artifact
    file_path         TEXT NOT NULL,
    content_hash      TEXT NOT NULL,
    materialized_at   TEXT NOT NULL DEFAULT (datetime('now')),
    PRIMARY KEY (epic_id, file_path)
);
```

Planning findings get a separate table because implementation `findings` has
`acceptance_criterion_id`, `location`, and implementation-specific
fingerprinting that does not apply to plans. Planning finding fingerprints use
`(category, target_type, target_sequence, description_prefix)` with the same
80-char prefix approach.

### Shared Tables

Reuse `agent_runs` for planner and planning-reviewer runs. The table already
has `role IN ('implementer','reviewer','planner','human')` and the run envelope
is role-agnostic. Planning reviewer runs use `role = 'reviewer'` (not a new
role enum value) and are distinguished from implementation reviewer runs by
`epic_id IS NOT NULL`. The role describes what the agent does; the FK target
describes which domain it operates in. Add a nullable `epic_id` column and
make `ticket_id` nullable with an XOR constraint:

```sql
ALTER TABLE agent_runs ADD COLUMN epic_id TEXT REFERENCES planned_epics(id);
-- Make ticket_id nullable and add:
CHECK (
    (ticket_id IS NOT NULL AND epic_id IS NULL) OR
    (ticket_id IS NULL AND epic_id IS NOT NULL)
)
```

Reuse `decisions` and `state_transitions` with the same approach — add nullable
`epic_id`, make `ticket_id` nullable, same XOR check. These tables are
structurally identical for both domains and separate tables would mean
duplicated queries for no benefit.

### Orchestrator Persistence

Generalize `orchestrator_state` with a `loop_type` discriminator rather than
adding a sibling table. A single-operator local tool runs one active loop at a
time — either planning or implementing, never both simultaneously. A sibling
table would duplicate the status/run tracking logic and force every consumer to
check two tables.

```sql
ALTER TABLE orchestrator_state
    ADD COLUMN active_plan_id TEXT REFERENCES planned_epics(id);
ALTER TABLE orchestrator_state
    ADD COLUMN loop_type TEXT CHECK (loop_type IN ('implementation', 'planning'));
```

Semantics:

- `status = 'idle'`: `loop_type` is NULL, `active_ticket_id` is NULL,
  `active_plan_id` is NULL
- `loop_type = 'implementation'`: `active_ticket_id` is set,
  `active_plan_id` is NULL
- `loop_type = 'planning'`: `active_plan_id` is set,
  `active_ticket_id` is NULL

### Lineage

Add a nullable FK to `tickets` for materialization lineage:

```sql
ALTER TABLE tickets
    ADD COLUMN planned_ticket_id TEXT REFERENCES planned_tickets(id);
```

One planned ticket materializes into one implementation ticket. A join table
would be over-engineering for a 1:1 relationship.

## Implementation Notes

- Keep planning entities distinct from implementation tickets. Approved plans
  may later materialize implementation tickets, but planning rows should not be
  overloaded into the current `tickets` table.
- Mirror the existing separation between canonical DB state, transition log,
  and append-only activity logging.
- Prefer additive migrations such as `0003_*` rather than editing historical
  migrations. SQLite does not support `ALTER COLUMN` or adding `CHECK`
  constraints to existing columns, so the migration will need to recreate
  `agent_runs`, `decisions`, and `state_transitions` with the new nullable
  `epic_id` columns and XOR checks. Use the standard
  create-new/copy/drop-old/rename pattern.
- If shared utilities in [src/capsaicin/state_machine.py](../../../src/capsaicin/state_machine.py)
  can be generalized cleanly for both planning and ticket transitions, do that;
  otherwise keep planning transition logic separate rather than forcing an
  awkward abstraction too early.
