# T01: Planning Data Model And State Machine Foundation

## Goal

Introduce the canonical planning-domain schema, state-machine rules, and query
primitives needed to support a local planning loop alongside the existing
implementation loop.

## Scope

- define planning entities and relationships in code and docs
- add migrations for planning records, planning findings, planning decisions,
  and planning transitions
- decide and implement the canonical orchestrator persistence model for
  planning runs
- define lineage storage between planning records and materialized
  implementation tickets
- add planning status enums and transition validation
- add low-level query helpers and shared IDs needed by later planning-loop
  services
- keep manual implementation-ticket creation working unchanged

## Non-goals

- planner prompt assembly
- planner/reviewer subprocess invocation
- CLI command wiring
- ticket materialization rendering
- UI pages

## Acceptance Criteria

- the schema introduces first-class planning records for at least:
  - a problem statement or planning brief
  - a planned epic
  - planned implementation tickets
  - planning review findings
  - planning decisions and transition history
- planning statuses and transition rules are defined and documented in
  [docs/state-machine.md](../../state-machine.md), including:
  `planning/new -> planning/drafting -> planning/in-review -> planning/revise -> planning/human-gate -> planning/approved` plus `planning/blocked`
- the migration resolves orchestrator persistence for planning explicitly:
  either generalize `orchestrator_state` safely for both loops or add a sibling
  planning-specific orchestrator table
- planning findings are stored in planning-specific tables rather than
  overloading the existing implementation `findings` table
- lineage required for later materialization is persisted, either via planning
  IDs on materialized implementation tickets or via an explicit join table
- migrations are additive and preserve existing implementation-loop projects
- foreign keys, indexes, and IDs are defined with the same rigor as the
  existing implementation-loop schema
- shared query helpers exist so later tickets do not need to hand-roll raw SQL
  for basic planning record loading
- docs are updated where the planning model is now concrete rather than merely
  aspirational, including the planning note in
  [docs/data-model.md](../../data-model.md)

## Implementation Notes

- Keep planning entities distinct from implementation tickets. Approved plans
  may later materialize implementation tickets, but planning rows should not be
  overloaded into the current `tickets` table.
- Use separate planning findings/decisions/transitions tables unless a shared
  abstraction is clearly cleaner without introducing nullable-polymorphic
  schema ambiguity. The default assumption for this epic is separate tables.
- Mirror the existing separation between canonical DB state, transition log,
  and append-only activity logging.
- Prefer additive migrations such as `0003_*` rather than editing historical
  migrations.
- Resolve the orchestrator-table question in this ticket, not later. T04 should
  build on the chosen schema rather than define it.
- Resolve materialization lineage in this ticket as well so T05 can rely on a
  concrete persistence model.
- If shared utilities in [src/capsaicin/state_machine.py](../../../src/capsaicin/state_machine.py)
  can be generalized cleanly for both planning and ticket transitions, do that;
  otherwise keep planning transition logic separate rather than forcing an
  awkward abstraction too early.
