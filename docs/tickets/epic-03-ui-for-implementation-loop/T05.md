# T05: Add narrow live updates with SSE

**Phase**: 4 — Live Updates
**Type**: Implementation
**Status**: Planned

## Goal

Add narrowly scoped live updates so the UI can reflect active runs and new
human-gate arrivals without a full SPA architecture.

## Scope

- add SSE endpoints for high-signal updates only
- stream updates for:
  - orchestrator active state
  - recent activity feed entries
  - ticket state changes relevant to dashboard and ticket detail
- connect the dashboard and ticket detail screens to those streams where it
  materially improves operator awareness
- degrade cleanly when SSE is unavailable or disconnected

## Non-goals

- no broad client-side state management layer
- no websocket architecture
- no attempt to make every page element real-time

## Acceptance criteria

- active run state and human-gate arrival can appear in the UI without manual
  refresh
- SSE usage is limited to specific operator-relevant surfaces
- the UI remains usable if the event stream is unavailable
- server-side event production does not require duplicating workflow logic

## Implementation Notes

- Prefer event payloads derived from existing read models or small identifiers
  that trigger partial re-rendering.
- Avoid building a second data API just for SSE.
- Keep connection handling simple and local-first. This is an operator
  convenience feature, not a distributed systems problem.
- Favor server-triggered partial re-rendering over pushing large custom client
  payloads.
