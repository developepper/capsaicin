# T04: Build ticket detail and diagnostics views

**Phase**: 3 — Read Models And Screens
**Type**: Implementation
**Status**: Planned

## Goal

Provide a detailed ticket view that exposes the state, findings, run history,
diagnostics, and diff context needed to understand and operate one ticket.

## Scope

- refine the initial ticket-detail and diagnostics read-model contracts from
  T01 so they support a real operator page and include:
  - ticket metadata and current state
  - acceptance criteria and statuses
  - open findings and recent dispositions
  - last-run diagnostics including cost and permission-denial summaries
  - run history
  - state transition history
  - reviewed diff or implementation diff summary
- implement ticket detail routes and templates
- support navigation from the dashboard into ticket detail
- make the human-gate rationale legible when present

## Non-goals

- no approve/revise/defer action workflow yet
- no full code-review UI beyond pragmatic diff display
- no ticket creation UI

## Acceptance criteria

- an operator can inspect one ticket end-to-end without using `capsaicin status`
- reliability diagnostics added in Epic 02 are visible in the ticket view
- the view distinguishes status, gate reason, blocked reason, findings, and run
  outcomes clearly
- the page uses shared read models rather than directly assembling SQL in
  templates
- the ticket view is structured so it can become the durable landing surface
  for long-running action redirects when those actions are introduced

## Implementation Notes

- Keep the initial diff presentation pragmatic. A readable plain diff block or
  compact file-change summary is acceptable if it is accurate and operator-useful.
- Avoid inventing HTML-specific diagnostic logic. Reuse the same structured
  diagnostic signals that back CLI status views.
- If the ticket-detail read model becomes large, split it into coherent nested
  sections instead of many loosely related helpers.
- Preserve the reliability and diagnostics semantics from Epic 02; this page is
  one of the main places operators will decide whether to approve, revise, or
  defer.
- This page should be designed to become the natural post-action destination
  for `run`, `review`, and `resume` once those triggers exist in the UI.
