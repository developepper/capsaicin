# T01: Extract UI-safe service and read-model boundaries

**Phase**: 1 — Shared Boundaries
**Type**: Implementation
**Status**: Planned

## Goal

Create shared application-service and read-model seams so both the CLI and the
future web UI can operate on the same structured contracts.

## Scope

- identify the workflow actions that should become shared service entry points
- extract service-level functions for:
  - ticket selection for run/review/approve/revise/defer/unblock/resume paths
  - run/review/approval/revision/defer/unblock execution
  - status and diagnostic retrieval needed by both CLI and UI
- extract structured read models for operator views instead of returning
  CLI-formatted strings
- update CLI commands to use the shared services and view models
- keep existing behavior and output semantics materially intact
- document the new boundary in architecture-facing docs where appropriate

## Non-goals

- no HTTP server
- no HTML templates
- no new UI output

## Acceptance criteria

- core operator actions have shared service functions that do not depend on
  Click or CLI output rendering
- project summary, ticket detail, diagnostics, inbox, and activity-oriented
  view contracts exist as structured data rather than only pre-rendered text
- CLI commands consume the shared contracts instead of reassembling the same
  data ad hoc
- no workflow progression requires shelling out to the CLI
- docs reflect the new service and read-model boundary clearly enough to guide
  later web work

## Dependencies

- None

## Implementation Notes

- Treat this ticket as architecture work, not a cosmetic refactor.
- Prefer explicit command/query style boundaries:
  - command-style services mutate workflow state and return structured outcomes
  - query-style read models return screen-ready data without side effects
- Avoid introducing framework-specific abstractions here. The boundary should
  remain useful if the future UI is not HTTP-based.
- It is acceptable for CLI text rendering helpers to remain, but they should
  consume structured view models rather than querying and formatting inline.
- Favor a small number of stable read models over a large number of narrow
  helper functions.
- Refine the module shape before editing code. A good initial cut is:
  - `application/commands` or equivalent for workflow mutations
  - `application/queries` or equivalent for view/read models
  - existing pipeline modules kept as implementation details behind command
    services where practical
- Recommended concrete package layout for this ticket:
  - `src/capsaicin/app/context.py`
  - `src/capsaicin/app/commands/run_ticket.py`
  - `src/capsaicin/app/commands/review_ticket.py`
  - `src/capsaicin/app/commands/approve_ticket.py`
  - `src/capsaicin/app/commands/revise_ticket.py`
  - `src/capsaicin/app/commands/defer_ticket.py`
  - `src/capsaicin/app/commands/unblock_ticket.py`
  - `src/capsaicin/app/commands/resume.py`
  - `src/capsaicin/app/commands/loop.py`
  - `src/capsaicin/app/queries/dashboard.py`
  - `src/capsaicin/app/queries/ticket_detail.py`
  - `src/capsaicin/app/queries/inbox.py`
  - `src/capsaicin/app/queries/activity.py`
  - `src/capsaicin/app/queries/diagnostics.py`
- Implement this ticket in two internal slices even if it lands as one PR:
  - Slice A: command services and CLI adoption
  - Slice B: initial read-model contracts and CLI/status adoption
- The first extraction target should be shared action entry points for:
  - run
  - review
  - approve
  - revise
  - defer
  - unblock
  - resume
  - loop
- The first read-model target should be structured payloads for:
  - project dashboard summary
  - ticket detail
  - human-gate inbox
  - recent activity / recent runs
  - run diagnostics
- T01 owns the initial contracts and shared query composition for those read
  models. T03 and T04 own the screen-specific refinement, template mapping, and
  any additions discovered while building real pages.
- Keep project-context resolution and config refresh as shared pre-handler logic
  that both CLI and UI can call without reimplementing discovery rules.
- `app/context.py` should centralize:
  - project discovery
  - config snapshot refresh
  - DB connection acquisition
  - lightweight request/command context objects shared by CLI and web handlers
- `app/commands/*` should return structured outcomes suitable for both CLI and
  web callers:
  - selected ticket or run identifiers when relevant
  - final status
  - diagnostic or redirect-relevant metadata
  - human-gate context where the caller needs to branch immediately
- `app/queries/*` should return serializable dictionaries or typed dataclasses
  that are screen-ready but not HTML-aware.
- Do not move core workflow logic blindly. Prefer wrapping the current modules
  first, then extracting shared pieces only when duplication remains.
- Testing guidance for this ticket:
  - keep the existing CLI and workflow tests passing as regression coverage
  - add focused tests for `app.commands/*` where they introduce new branching,
    context handling, or structured outcome contracts
  - add focused tests for `app.queries/*` around returned shapes and key status
    / diagnostic cases
  - it is acceptable to migrate some assertions out of CLI-oriented tests into
    service/query-layer tests when that improves boundary clarity, but do not
    reduce behavioral coverage overall
- Suggested implementation order:
  1. add `app/context.py` and shared context helpers
  2. add `app/commands/*` wrappers for existing workflow actions
  3. migrate `cli.py` to the new command layer without changing behavior
  4. add `app/queries/diagnostics.py` and `app/queries/activity.py`
  5. extract `app/queries/dashboard.py`, `ticket_detail.py`, and `inbox.py`
  6. update CLI status rendering to consume the new query layer
- Keep `ticket_status.py` as a temporary compatibility layer if that reduces
  churn, but it should end T01 as a renderer over `app.queries`, not as the
  place where read-model logic lives.
- Keep imports one-directional where possible:
  - `cli.py` -> `app.context`, `app.commands`, `app.queries`
  - `web/*` -> `app.context`, `app.commands`, `app.queries`
  - `app.commands/*` -> existing workflow modules
  - `app.queries/*` -> DB/query helpers and diagnostics helpers
  - existing workflow modules should not depend on `web/*`
