# T06: Implement the human-gate inbox and decision flow

**Phase**: 5 — Human-Gate Flow
**Type**: Implementation
**Status**: Planned

## Goal

Make the human-gate path first-class in the UI so operators can inspect the
relevant context and take approve, revise, defer, and unblock actions safely.

## Scope

- add a human-gate inbox view or dashboard section with clear prioritization
- add action forms for:
  - approve
  - revise
  - defer
  - unblock where appropriate
- add UI action triggers for `run`, `review`, `resume`, and `loop` if they were
  not already added in T03/T04
- surface rationale requirements and validation for override-like approvals
- show the reviewed-state mismatch or drift constraints where they matter
- redirect or refresh the relevant ticket/dashboard surfaces after decisions

## Non-goals

- no PR creation workflow
- no planning-loop decisions
- no multi-user review/approval coordination

## Acceptance criteria

- an operator can complete the current human-gate workflow from the UI
- an operator can initiate the main implementation-loop actions from the UI,
  either because they landed here or in an earlier screen ticket
- required rationale and safety checks match the existing CLI and service
  behavior
- human decisions persist through the same shared services used by the CLI
- UI messaging makes it clear why a ticket is at the gate and what each action
  will do next

## Implementation Notes

- Keep action semantics identical to the shared services. The web UI should not
  soften or reinterpret state-machine rules.
- Approve/revise/defer are not symmetric from an operator-risk perspective.
  The UI should reflect that instead of presenting them as equal-weight buttons.
- Make the rationale and gate-reason context visible before the operator acts.
- Include the same safety constraints the CLI enforces for rationale-required
  approvals and reviewed-workspace mismatches.
- If `run`, `review`, `resume`, or `loop` are still missing when this ticket
  starts, add them here using the redirect-to-live-status pattern established by
  T03/T04/T05.
