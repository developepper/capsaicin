# T13: Claude Code adapter — reviewer mode

**Phase**: 4 — Adapter Layer
**Type**: Implementation
**Suggested PR pairing**: Ship with T12 in a single PR (same class, two modes)

## Goal

Invoke Claude Code as a reviewer with structured output extraction and
validation.

## Scope

- Extend `ClaudeCodeAdapter` with reviewer execution path
- Command: `claude -p --output-format json --json-schema <SCHEMA>
  --allowed-tools <TOOLS...> -- "PROMPT"`
- Pass full MVP Review Result Schema from `adapters.md` via `--json-schema`
- Pass prompt after `--` to avoid tool-name parsing
- Parse `structured_output` from envelope as primary structured result
- Fall back to `result` only if `structured_output` absent
- Run semantic validation after JSON schema extraction (import from T17's
  `validate_review_result`):
  - `verdict:fail` requires >=1 blocking finding
  - `verdict:pass` cannot have blocking findings
  - `confidence:high` invalid with empty `files_examined`
  - `confidence:high` invalid with criteria provided but `criteria_checked` empty
  - `criteria_checked` entries must reference valid criterion IDs
  - `acceptance_criterion_id` on findings must reference valid criterion IDs
- On validation failure: `exit_status=parse_error`, raw output preserved
- Tests: captured real Claude Code reviewer JSON envelopes in `tests/fixtures/`;
  mocked subprocess tests for process behavior
- Ensure reviewer fixture tests run in the existing CI workflow via `pytest`
  without requiring Claude authentication or network access

## Non-goals

- No review baseline checking
- No finding reconciliation

## Acceptance criteria

- Reviewer invocation includes `--allowed-tools` from config
- Prompt passed after `--`
- Structured result read from `structured_output` field
- Valid ReviewResult passes validation
- Each semantic validation rule produces `parse_error` when violated
- Raw output preserved on any parse_error
- Reviewer fixture and semantic-validation tests pass in CI
