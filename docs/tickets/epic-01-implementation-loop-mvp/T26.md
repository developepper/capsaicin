# T26: `capsaicin resume`

**Phase**: 8 — Status & Recovery
**Type**: Implementation
**Risk**: Depends on T15/T20 pipeline logic being well-factored

## Goal

Recover from interrupted execution.

## Scope

- `capsaicin resume`
- Read `orchestrator_state`
- `idle`: behave like `ticket run` (auto-select next ready ticket)
- `running` + run has `finished_at`: execute the post-run pipeline for the run's
  role (re-use the post-run logic from T15 or T20)
- `running` + run has no `finished_at`: interrupted — mark run as `failure`,
  increment retry or block
- `awaiting_human`: render human-gate context (ticket, gate_reason, findings,
  criteria) and stop
- `suspended`: use `resume_context` to determine which step to continue

## Non-goals

- No new state machine transitions beyond T08

## Acceptance criteria

- `idle` triggers ticket selection and run
- Finished-but-unprocessed implementation run completes post-run pipeline
- Finished-but-unprocessed review run completes post-run pipeline
- Interrupted run (no `finished_at`) -> retry or block
- `awaiting_human` renders context without action
- Resume after crash with completed run does not duplicate work

## Dependencies

- T15
- T20

## Notes

Resume correctness depends on T15 and T20 factoring post-run pipeline logic
into reusable functions. If those tickets embed logic in click handlers, this
ticket will force a refactor.
