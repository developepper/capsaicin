# T02: Empty-implementation diagnostics

**Phase**: 2 — Empty-Run Diagnostics
**Type**: Implementation

## Goal

Make empty implementation outcomes explain themselves in human-gate and related
operator messaging.

## Scope

- improve inline stop/output messaging for `ticket run`, `loop`, and
  `resume`-driven human-gate rendering
- extract agent result text from persisted `raw_stdout` at display time rather
  than adding a new `agent_runs.result_text` column
- when available, include a truncated agent result text snippet in the message
- when the run was blocked by permissions, say so explicitly and include a
  summary count of denied actions
- include the high-signal denied tool names in operator-facing output when that
  adds clarity
- ensure operator-facing wording distinguishes:
  - no changes made
  - blocked by permissions
  - other execution failures already handled elsewhere

## Non-goals

- no new `doctor` command
- no activity log changes
- no retrospective `status` display changes beyond whatever is required to keep
  shared human-gate context helpers coherent

## Acceptance criteria

- empty implementation messages surface agent result text when it is useful
- permission-blocked runs explicitly mention denied actions rather than reading
  as plain empty implementations
- short remediation text such as "grant write permission" is preserved in the
  surfaced message when the agent already provided it
- human-gate context built from persisted runs can render the same explanation
  without requiring direct database inspection
- loop stop messages and direct `ticket run` output are consistent
