# T01: Distinct permission-denied run outcomes

**Phase**: 1 — Adapter Signals
**Type**: Implementation

## Goal

Detect Claude Code permission denials and classify blocked runs distinctly from
successful empty implementations.

## Scope

- inspect `permission_denials` in the Claude Code envelope for implementer and
  reviewer runs
- add a distinct run outcome for permission-blocked execution
- route permission-blocked implementer and reviewer runs directly to
  `human-gate` without consuming retries
- add `gate_reason = 'permission_denied'` for this path and update schema
  constraints accordingly
- normalize denial entries into a stable internal shape with:
  - `tool_name`
  - `tool_use_id`
  - `file_path` for `Edit` and `Write` when present
  - `command` for `Bash` when present
- persist denial details under a normalized adapter metadata key while
  preserving the raw `permission_denials` envelope data
- update schema, types, and persistence paths as needed for the new outcome,
  including:
  - `adapters/types.py` valid exit-status definitions
  - `agent_runs.exit_status` database constraint
  - `tickets.gate_reason` database constraint
  - shared run-result docs and type contracts, including:
    - `docs/adapters.md`
    - `docs/data-model.md`
    - `docs/state-machine.md`
    - `docs/cli.md`
- add explicit branches for the new outcome in implementation and review
  post-run handlers, including resume paths
- copy the captured permission-denied fixtures into `tests/fixtures/`
- tests for denied Edit/Write calls on mocked and fixture-backed envelopes

## Non-goals

- no human-gate rendering changes
- no preflight checks

## Acceptance criteria

- a run with one or more permission denials is not classified as plain
  `success`
- denial details include the denied tool and target path when available
- repeated denials are preserved as distinct attempts rather than silently
  deduplicated
- implementer and reviewer permission-denied runs transition to `human-gate`
  with `gate_reason='permission_denied'` and do not consume retries
- shared post-run handlers and resume paths understand the new run outcome
- run records persist the new outcome without schema violations
- database constraints and type-level exit-status validation accept the new
  value
- docs are updated anywhere the run outcome and gate reason are part of the
  public contract, including `docs/adapters.md`, `docs/data-model.md`,
  `docs/state-machine.md`, and `docs/cli.md`
- fixture and unit tests cover implementer and reviewer envelopes with
  permission denials

## Implementation Notes

- Review the captured permission-denied fixtures before changing adapter logic:
  - `/tmp/capsaicin-fixtures/claude_envelope_permission_denied_edit_only.json`
  - `/tmp/capsaicin-fixtures/claude_envelope_permission_denied_mixed.json`
- Copy those fixtures into `tests/fixtures/` as part of this ticket so they are
  durable project assets rather than ephemeral `/tmp` files.
- Preserve the full Claude envelope in `raw_stdout`; normalize denial details in
  adapter metadata under a new normalized key without discarding raw evidence.
- Update shared contracts coherently across adapter types, persistence, and any
  schema checks. Avoid a half-state where the adapter emits a new outcome that
  the database or CLI cannot represent.
- SQLite CHECK constraint changes likely require table recreation in migration
  logic. Treat that as part of the schema work, not as an incidental detail.
- Reviewer permission denials are expected to be rare because reviewer runs are
  read-only, but they should use the same long-term model: first-class run
  outcome, no automatic retries, and transition to `human-gate` for operator
  action.
- Keep CI deterministic. Use fixtures and mocked subprocess behavior; do not
  require live Claude access.
- Follow existing project conventions: Python 3.11, `click`, SQLite, ULIDs,
  and Ruff formatting.
