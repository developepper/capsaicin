# T06: Loop auto-selection should include revise tickets

**Phase**: 4 — Loop And Status UX
**Type**: Implementation

## Goal

Make `capsaicin loop` continue in-flight revise work without requiring an
explicit ticket ID.

## Scope

- update no-argument `capsaicin loop` ticket selection
- prefer runnable `revise` tickets before unrelated `ready` work
- preserve cycle-limit shortcut behavior and existing dependency rules
- add tests covering ready-only, revise-only, and mixed queues

## Non-goals

- no multi-ticket orchestration
- no changes to manual `ticket run` selection rules unless required for shared
  helper consistency

## Acceptance criteria

- `capsaicin loop` with no ticket ID selects a revise ticket when one is
  available
- revise selection still respects cycle-limit handling
- mixed queues behave predictably and do not require manual ticket IDs to
  continue the active loop

## Dependencies

- None

## Implementation Notes

- Read the existing no-argument selection behavior in `ticket run`, `resume`,
  and `loop` before changing queue logic. The goal is to continue in-flight
  revise work without breaking existing ready-ticket selection semantics.
- Prefer a shared selection helper if the codebase already has one or clearly
  wants one; avoid divergent queue rules across commands unless intentional.
- Preserve dependency checks and cycle-limit shortcuts exactly as documented.
- Add tests that cover the user-observed confusing case: revise ticket exists,
  no ready ticket exists, and `loop` should still proceed.
