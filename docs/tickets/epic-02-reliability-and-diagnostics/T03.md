# T03: Run-cost and denial logging

**Phase**: 2 — Empty-Run Diagnostics
**Type**: Implementation

## Goal

Record high-signal run diagnostics in `activity.log` so operator-visible traces
match what actually happened.

## Scope

- include `total_cost_usd` in run-end log payloads when available
- log permission denials explicitly with tool and target path details
- make run-end logging reflect the final classified run outcome
- keep the log line format stable while extending only the JSON payload
- ensure real clean exits with `permission_denials` do not log as plain
  successful empty runs

## Non-goals

- no additional rendered reports
- no budgeting or aggregated cost reporting

## Acceptance criteria

- `activity.log` run-end entries include cost when the adapter reports it
- permission denials are visible in the log without reading raw envelopes
- denial entries capture `file_path` for edit-like tools and `command` for bash
  tools when present
- log payload shape remains compact and machine-readable

## Implementation Notes

- Extend the JSON payload in `activity.log` without breaking the existing line
  format documented in `architecture.md`.
- Prefer high-signal fields only: classified outcome, cost, denied tool, denied
  path or command, and concise counts. Avoid logging full prompts or tool
  payloads into `activity.log`.
- Use the same normalized denial shape introduced in T01 so logging, status, and
  human-gate messaging do not invent separate interpretations.
- Preserve deterministic tests. Logging tests should assert structured payload
  content, not wall-clock-specific values beyond what is already standardized.
