# T05: Plan Approval And Implementation-Ticket Materialization

## Goal

Turn an approved plan into concrete implementation-ticket documents and local
records that the existing implementation loop can use without requiring hidden
chat context.

## Scope

- add human approval and defer/revise actions for planning records
- define the renderer/exporter that materializes approved plans into
  implementation-ticket docs
- create or update the local implementation-loop ticket DB records from
  approved planned tickets
- establish lineage between planning artifacts and materialized implementation
  tickets
- ensure generated ticket docs include references, dependencies, and
  implementation notes

## Non-goals

- `gh` issue creation
- PR automation
- frontend UI work beyond whatever read models are necessary for later tickets

## Acceptance Criteria

- an approved plan can be materialized into implementation tickets in a
  deterministic way
- materialization creates both:
  - implementation-ticket markdown docs under `docs/tickets/` for session
    context
  - implementation-loop `tickets` / dependency / criteria records in the local
    DB so the existing loop can execute them
- generated implementation-ticket docs are sufficient for a fresh coding
  session and explicitly list required references and implementation notes
- dependencies between planned tickets are preserved in the materialized
  implementation records
- the system records lineage from planning epic/ticket IDs to implementation
  ticket IDs
- re-materialization rules are documented and tested so operator edits are not
  silently destroyed
- manual ticket creation still works for projects that do not use the planning
  loop

## Design Decisions

### Materialization Location

Generated implementation-ticket docs go under `docs/tickets/generated/<epic-slug>/`
to keep a clear boundary between planning-stage docs (such as the planning epic
breakdowns under `docs/tickets/epic-*/`) and executable implementation-ticket
docs produced by materialization. The slug is derived from the epic title
(lowercase, hyphens, strip special chars).

```text
docs/tickets/generated/<epic-slug>/
  README.md       # epic overview
  T01.md          # ticket with sequence 1
  T02.md          # ticket with sequence 2
  ...
```

The output path is stored on `planned_epics.materialized_path` so the system
knows where generated docs live.

### Regeneration Policy

Content-hash gating with three outcomes. At materialization time, a SHA-256
hash of each generated file is stored in `materialization_hashes` (defined in
T01), keyed by `(epic_id, file_path)` with an optional `planned_ticket_id` for
ticket-level artifacts. Epic-level artifacts such as README.md have
`planned_ticket_id = NULL`. On re-materialization:

| File state | Hash matches current file? | Action |
|---|---|---|
| File does not exist | n/a | Write it |
| File exists, hash matches | Yes | Overwrite (no manual edits detected) |
| File exists, hash differs | No | Refuse and report the conflict |

When a file is refused, the operator gets a message such as: "T03.md has been
manually edited since materialization. Delete it or pass `--force` to
overwrite." The `--force` flag overwrites and updates the stored hash.

This avoids silent destruction of manual edits while still supporting clean
re-materialization after plan revisions.

## Implementation Notes

- Materialized implementation tickets should live under `docs/tickets/` in a
  clearly documented location for active work. Do not hide the handoff inside
  the database only.
- The markdown docs are not supplementary decoration. They are the human-facing
  session artifact, while the DB rows remain the executable source for the
  implementation loop. Materialization must create and link both.
- Generated ticket docs must follow the authoring convention from
  [docs/tickets/README.md](../../README.md): goal, scope, non-goals, acceptance
  criteria, dependencies, references, and implementation notes.
- Treat materialization as an explicit approval-side effect, not something that
  happens during drafting or review.
- T01 must provide the lineage persistence model and `materialization_hashes`
  table that this ticket writes to.
