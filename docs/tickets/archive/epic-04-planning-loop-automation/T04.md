# T04: Planning Loop Orchestration And Resume

## Goal

Implement the bounded planning draft/review/revise loop and resume behavior so
`capsaicin` can drive planning until a human decision or blocker is reached.

## Scope

- add planning-loop orchestration analogous to the existing implementation loop
- persist active planning run state in the orchestrator layer
- enforce cycle and retry rules for planning draft/review passes
- support resume after interrupted planning runs
- emit activity-log events for planning-loop execution

## Non-goals

- UI work
- final ticket materialization rendering
- GitHub issue creation

## Acceptance Criteria

- a dedicated `capsaicin plan loop` command can draft, review, revise, and
  stop at `planning/human-gate` or `planning/blocked`
- cycle limits route planning work to human review rather than infinite
  planner/reviewer churn
- retry limits distinguish transient execution problems from review findings
- resume can safely recover interrupted planner or planning-reviewer runs using
  persisted orchestrator state
- activity logging and state transitions make planning-loop execution
  inspectable for operators
- tests cover at least: clean pass to human gate, fail to revise, cycle-limit
  gate, retry-limit block, and interrupted-run resume

## Implementation Notes

- T01 must already have resolved whether planning shares `orchestrator_state`
  or uses a sibling table. This ticket should implement against that chosen
  persistence model rather than redefine it.
- Keep the planning loop bounded and inspectable. Fresh-session review is still
  load-bearing here.
- The planning loop should stop before materialization. Human approval remains
  the boundary between “candidate plan” and “approved implementation work.”
