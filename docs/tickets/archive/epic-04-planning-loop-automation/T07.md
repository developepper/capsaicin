# T07: Explicit Implementation Completion From `pr-ready` To `done`

## Goal

Add an explicit operator action to complete an implementation ticket after it
reaches `pr-ready`, so dependent tickets can advance without forcing a specific
Git or GitHub workflow.

## Scope

- add a CLI command to mark a `pr-ready` ticket as `done`
- add a matching UI action on `pr-ready` tickets
- record the human completion decision and rationale when provided
- update dependency progression so downstream tickets can advance once a human
  explicitly completes the upstream ticket
- keep the future GitHub/PR automation path free to reuse this transition

## Non-goals

- automatic Git commit creation
- automatic PR creation
- GitHub integration
- changing dependency satisfaction rules so `pr-ready` counts as equivalent to
  `done`

## Acceptance Criteria

- there is an explicit command for transitioning `pr-ready -> done`
- the UI exposes the same completion action for `pr-ready` tickets
- the completion flow records a human decision in persistent state
- the transition is validated through the same state-machine boundaries as
  other ticket actions
- once a ticket is marked `done`, dependent tickets can become runnable
- docs clearly explain the difference between `pr-ready` and `done`

## Dependencies

- none

## Implementation Notes

- `pr-ready` should remain the explicit handoff state meaning “approved and
  ready for the developer's downstream workflow.”
- `done` should remain a separate explicit completion state meaning “the human
  considers the downstream handoff complete enough to unblock dependents.”
- This ticket should not assume any specific downstream workflow. For some
  operators that means “I committed and opened the PR myself.” For future
  automation it may mean “GitHub handoff completed successfully.”
- Prefer a command such as `capsaicin ticket complete [TICKET_ID]` rather than
  overloading `approve`.
- The UI action should only appear for `pr-ready` tickets.
- When GitHub support is added later, that automation can call the same
  transition or command-service boundary instead of inventing a parallel
  completion path.
