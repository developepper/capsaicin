# T06: Planning UI Surfaces

## Goal

Extend the local web UI so operators can inspect and drive the planning loop
with the same local-first model used for the implementation loop UI.

## Scope

- add planning dashboard and detail views
- add planning action forms for draft, review, revise, approve, defer, and
  resume/loop actions as appropriate
- surface planning findings, transitions, and materialization status
- add SSE-driven live updates for planning views where useful
- keep UI actions delegating to shared app command services

## Non-goals

- building a separate frontend app
- remote access or authentication
- GitHub integration screens

## Acceptance Criteria

- operators can inspect active planning work from the browser without losing
  parity with the CLI
- planning detail pages show the current plan draft, planned tickets, findings,
  transitions, and human-gate context
- UI actions use the same state-machine and command boundaries as the CLI
- the UI clearly distinguishes planning records from implementation tickets
- tests cover core planning routes and action-post behavior

## Implementation Notes

- Preserve the current local-only Starlette/Jinja/HTMX model. Planning UI
  should extend existing surfaces rather than introducing a new runtime.
- Keep planning and implementation views visually and structurally distinct so
  operators can tell whether they are shaping work or executing it.
- Only add SSE where it materially improves operator awareness; avoid broad
  redraw behavior when a narrower partial update is enough.
