# T06: Loop auto-selection should include revise tickets

**Phase**: 4 — Loop And Status UX
**Type**: Implementation
**Status**: Complete

## Goal

Make `capsaicin loop` continue in-flight revise work without requiring an
explicit ticket ID.

## Scope

- update no-argument `capsaicin loop` ticket selection
- prefer `revise` tickets before unrelated `ready` work using a loop-specific
  selection rule rather than silently changing `ticket run` auto-selection
- when multiple `revise` tickets exist, order them by `status_changed_at`
  ascending, then `created_at` ascending
- treat `revise` tickets as already in-flight work and do not re-check
  dependency satisfaction before selection
- preserve cycle-limit shortcut behavior and existing dependency rules
- keep `resume` in `idle` state aligned with `ticket run` behavior rather than
  adopting loop-specific revise selection
- add tests covering ready-only, revise-only, and mixed queues

## Non-goals

- no multi-ticket orchestration
- no changes to manual `ticket run` selection rules unless required for shared
  helper consistency

## Acceptance criteria

- `capsaicin loop` with no ticket ID selects a revise ticket when one is
  available
- revise selection still respects cycle-limit handling
- `ticket run` auto-selection behavior remains unchanged unless the
  implementation introduces an explicitly parameterized shared helper
- `resume` in `idle` state remains aligned with `ticket run` rather than
  preferentially selecting `revise` tickets
- mixed queues behave predictably and do not require manual ticket IDs to
  continue the active loop

## Implementation Notes

- Read the existing selection helpers first. In the current implementation,
  `select_ticket()` is a likely touchpoint for `loop` and `resume`, but this
  ticket should not accidentally broaden `ticket run` auto-selection unless
  that is made explicit by design.
- Prefer a loop-specific selector or a parameterized shared helper so the
  selection semantics stay intentional across commands.
- If a shared helper is parameterized, keep the default semantics aligned with
  `ticket run` and opt `loop` into revise-first behavior explicitly.
- Preserve dependency checks and cycle-limit shortcuts exactly as documented.
- Add tests that cover the user-observed confusing case: revise ticket exists,
  no ready ticket exists, and `loop` should still proceed.
