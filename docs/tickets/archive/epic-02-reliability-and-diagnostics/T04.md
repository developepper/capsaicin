# T04: Preflight check module

**Phase**: 3 — Preflight Checks
**Type**: Implementation
**Status**: Complete

## Goal

Create reusable environment and repo validation helpers for operator-facing
setup checks.

## Scope

- add a module that validates:
  - adapter command availability on PATH
  - configured `repo_path` exists
  - configured repo is a git repository
  - working tree cleanliness or dirty-state warning
  - Claude local permission configuration for required write tools
- return structured check results with pass, warn, or fail status
- isolate repo-specific permission probing so it can be reused by `init`,
  `doctor`, or later commands
- read `<repo>/.claude/settings.local.json` when present and verify
  `permissions.allow` contains bare-string entries for `Edit` and `Write`

## Non-goals

- no CLI command yet
- no automatic mutation of Claude settings

## Acceptance criteria

- checks return structured results rather than free-form text only
- dirty working tree is a warning, not an unconditional failure
- missing Claude write permissions produce a clear failure or warning result
- Bash permission patterns are not validated beyond presence of the settings
  file because required Bash allow-list patterns are project-specific and not
  safely inferable from generic config
- permission checks understand the observed settings shape where Bash entries
  may be patterned strings like `Bash(git:*)` while `Edit` and `Write` are bare
  tool names
- `Write` is checked proactively alongside `Edit` even if current fixtures only
  demonstrate `Edit` denials
- tests cover missing command, invalid repo path, non-git repo, dirty repo, and
  missing permission configuration

## Implementation Notes

- Keep the check logic reusable and side-effect free. `doctor`, `init`, and any
  future setup validation should be able to call the same functions.
- Parse `.claude/settings.local.json` defensively. Missing files, invalid JSON,
  or absent keys should produce explicit structured results rather than
  unhandled exceptions.
- Distinguish `fail` from `warn` carefully. Missing required write permissions is
  high-signal; a dirty tree should usually warn.
- Prefer structured check-result objects over ad hoc dicts so later commands and
  a future UI can reuse them cleanly.
- Do not mutate repo configuration in this ticket.
