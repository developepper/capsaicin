# T05: UI for viewing and editing role/backend assignments

## Goal

Add UI panels on epic detail and ticket detail pages that display the resolved backend for each role and allow the operator to set or clear role-scoped overrides (planner/planning_reviewer at epic level, implementer/reviewer at ticket level), so role-to-backend mapping is transparent and editable.

## Scope

- Add a 'Role Assignments' panel to the epic detail page showing resolved backend for planner and planning_reviewer roles with the resolution source indicated (e.g., 'codex (epic override)' vs 'claude-code (project config)')
- Add a form on the epic detail page to set or clear an override for planner or planning_reviewer roles only: role selector (limited to planner/planning_reviewer), backend name, command, optional model and allowed_tools
- Add a similar 'Role Assignments' panel to the ticket detail page (for implementation tickets) showing resolved backend for implementer and reviewer roles with source, and a form to set/clear ticket-level overrides for implementer/reviewer only
- Add POST routes: /epics/{epic_id}/overrides (set override for planner/planning_reviewer), /epics/{epic_id}/overrides/{override_id}/delete (clear)
- Add POST routes: /tickets/{ticket_id}/overrides (set override for implementer/reviewer), /tickets/{ticket_id}/overrides/{override_id}/delete (clear)
- Use resolve_all_roles convenience function from ticket 3 to populate the display

## Non-goals

- Setting epic-level overrides for implementer/reviewer roles — these are ticket-scoped only
- Setting ticket-level overrides for planner/planning_reviewer roles — these are epic-scoped only
- Override validation against available backends — the operator is trusted to enter valid backend names that are in the registry
- Bulk override editing across multiple epics or tickets
- Planned ticket-level overrides in the UI

## Acceptance Criteria

- The epic detail page shows the resolved backend for planner and planning_reviewer roles with the source level (override vs config vs fallback) clearly indicated
- An operator can set an epic-level override only for planner or planning_reviewer roles from the epic detail page
- An operator can clear an epic-level override, causing resolution to fall back to project config
- The ticket detail page shows the resolved backend for implementer and reviewer with source level indicated
- An operator can set and clear ticket-level overrides for implementer and reviewer roles from the ticket detail page
- The role selector on the epic override form is limited to planner and planning_reviewer; the role selector on the ticket override form is limited to implementer and reviewer
- Changes to overrides are reflected immediately on page reload

## Implementation Notes

- The Role Assignments panel should be a compact table: Role | Backend | Command | Source — where Source is a badge like 'epic override', 'project config', or 'fallback'
- Follow the existing HTMX POST-and-redirect pattern for override create/delete
- For the epic detail page, show only planner and planning_reviewer roles (implementer/reviewer are ticket-scoped). For the ticket detail page, show only implementer and reviewer.
- The epic detail page can also show a read-only summary of the default implementer/reviewer config from project config for context, but override forms are only for planner/planning_reviewer
- Add a partials/role_assignments.html template partial that can be included from both epic and ticket content templates with different parameters (roles list and scope)
- The role selector dropdown on the epic form should only offer 'planner' and 'planning_reviewer' options. The ticket form should only offer 'implementer' and 'reviewer' options. This enforces the scoping model from ticket 3 at the UI level.
- The resolve_all_roles function from ticket 3 returns both the AdapterConfig and the source level string for display
